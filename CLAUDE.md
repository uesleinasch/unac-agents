# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Repository overview

This is a monorepo with two packages and a collection of agent/skill definitions:

- `agents-claude/` вЂ” source of truth for the atomic Claude Code agents (`.md`)
- `agents-vscode/` вЂ” source of truth for the VS Code / Copilot agents (`.agent.md`)
- `skills-shared/` вЂ” content skills used by both IDEs (`clean-code`, `api-patterns`, `jira-card-maker`, etc.)
- `skills-claude/` вЂ” Claude Code-only orchestration skills (`unac-pipeline`, `unac-execute-plan`, `unac-review-implementation`, `unac-fix-blockers`, `unac-multirepo`, `unac-explore`, `unac-verify-review`)
- `packages/unac-agents/` вЂ” npm-publishable CLI installer (`npx unac-agents`)
- `packages/interactive-mcp/` вЂ” MCP server for interactive user input (TypeScript + Ink)

**Never edit files inside `packages/unac-agents/assets/`** вЂ” that directory is generated by `npm run prepare` from the `agents-claude/`, `agents-vscode/`, `skills-shared/`, and `skills-claude/` sources.

## Common commands

### Syncing assets (after editing agents or skills)
```bash
cd packages/unac-agents
npm run prepare
```

### Building the MCP server
```bash
cd packages/interactive-mcp
npm run build         # tsc + tsc-alias
```

### Testing the installer locally
```bash
node packages/unac-agents/bin/install.js
```

### Publishing a new version
```bash
cd packages/unac-agents
npm version patch     # or minor / major
npm run release       # npm publish
```

## Agent pipeline architecture

Agents are atomic workers designed to run in sequence with explicit handoffs:

```
unac-product-owner в†’ unac-jira-maker в†’ unac-solution-architect в†’ unac-tech-lead
  в†’ unac-developer в†’ unac-qa-engineer в†’ unac-code-reviewer в†’ unac-code-fix
```

Two additional read-only workers support the pipeline: `unac-explorer` and `unac-finding-verifier` (both in `agents-claude/`).

The `unac-pipeline` skill (in `skills-claude/`) is the meta-orchestrator that drives the whole flow with human gates at the critical edges. The execution loops are also skills: `unac-execute-plan` (dispatches `unac-developer` once per task), `unac-review-implementation` (dispatches `unac-code-reviewer` once per task), and `unac-fix-blockers` (dispatches `unac-code-fix` once per рџ”ґ BLOCKING issue). **Workers never spawn workers** вЂ” a skill is the only dispatch point, so each task runs in an isolated subagent with no inherited context, avoiding context-window exhaustion. **Parallelism is allowed only in read-only fan-out**: in Phase 1, `unac-explore` dispatches `unac-explorer` agents in parallel to gather codebase facts and `unac-finding-verifier` agents to adversarially challenge those findings (writing `## Fatos nГЈo-confirmados / contestados` into `{item-id}_codebase-context.md`). In Phase 7, `unac-review-implementation` invokes `unac-verify-review` to adversarially verify each рџ”ґ BLOCKING finding before declaring the overall result (writing `## VerificaГ§ГЈo adversarial` into `{item-id}_code-review-report.md`); only surviving blockers feed Gate E and Phase 7.5.

The flow is **spec-driven and test-first**: a global `.unac/constitution.md` (project principles) governs architect, tech-lead, developer, reviewer and code-fix; acceptance tests are authored from the `jira-card` ACs and confirmed *failing* (Fase 4.5 + Gate C.5) *before* any implementation; `unac-execute-plan` then drives the Green phase and enforces acceptance-test immutability via a hash check. The `jira-card` is the **single source of truth** for acceptance tests вЂ” never the plan's per-task criteria. A Traceability Matrix (AC do card в†’ task в†’ test) is validated at Gate C.

When the product-owner detects that an item spans more than one repository, the pipeline switches to **multi-repo mode**: it confirms with the user, discovers the repos (scans the parent dir for `.git`, else asks), writes `.unac/{item-id}/workspace.md`, and delegates Phases 4.5вЂ“7.5 to the `unac-multirepo` skill. That skill runs the per-repo cycle (Redв†’Greenв†’QAв†’Review) **contract-first** (provider before consumer, anchored on `{item-id}_contract.md`), reusing `unac-execute-plan`/`unac-review-implementation`/`unac-fix-blockers` with a `repo-path`. Per-repo execution artefacts are suffixed (`..._<repo>.md`) and each repo has its own `.unac/constitution.md`. Single-repo remains the default, unchanged.

### Canonical `.unac/` artefacts

Each agent writes its artefacts to `.unac/{item-id}/` using the canonical naming convention `{item-id}_<name-in-kebab>.md` (e.g., `PROJ-123_implementation-plan.md`, `PROJ-123_qa-report.md`). The full canonical list вЂ” which artefact each phase creates and which agents read or edit it вЂ” is the source of truth in `skills-claude/unac-pipeline/SKILL.md` ("Artefatos canГґnicos de `.unac/`"). **Do not create artefacts outside that list** вЂ” no placeholders, no ad-hoc names. The `.unac/{item-id}/` directory is created implicitly by the first `Write` (the product-owner's `{item-id}_codebase-context.md`). There is also one **global** artefact вЂ” `.unac/constitution.md` вЂ” bootstrapped by the pipeline (Fase 0) and populated by the product-owner; it is not item-scoped.

## `packages/interactive-mcp` architecture

The MCP server exposes five tools registered in `src/index.ts`:

| Tool | Mode | Description |
|------|------|-------------|
| `request_user_input` | Single-shot | Spawns a terminal popup, waits for answer |
| `message_complete_notification` | Single-shot | Notification with optional reply |
| `start_intensive_chat` | Session | Opens a persistent chat window |
| `ask_intensive_chat` | Session | Sends a question into an open session |
| `stop_intensive_chat` | Session | Closes the session |

**Key data flow:**

1. `src/index.ts` вЂ” registers tools with MCP SDK; routes calls to command handlers
2. `src/tool-definitions/` вЂ” each tool's Zod schema (`schema`), capability metadata (`capability`), and description string/factory (`description`)
3. `src/commands/input/index.ts` вЂ” single-shot mode: writes options to a temp JSON file, spawns `commands/input/ui.tsx` as a child process, polls an output file for the response
4. `src/commands/intensive-chat/index.ts` вЂ” session mode: spawns `commands/intensive-chat/ui.tsx`, communicates via files in a temp `outputDir`; questions arrive as `{sessionId}.json`, responses as `response-{questionId}.txt`
5. `src/components/InteractiveInput.tsx` вЂ” the shared Ink/React component used by both UIs; handles arrow-key selection of `predefinedOptions`, typing for custom input, and the special "Outros" option (selecting it switches to text input mode instead of submitting)

> `src/ui/interactive-input.tsx` is **not imported anywhere** вЂ” it is a standalone draft/alternative version. Do not use it as a reference for the canonical component.

**Path alias:** `@/*` resolves to `src/*` (configured in `tsconfig.json` via `tsc-alias`).

## Switchboard protocol (cross-IDE agent collaboration)

The `.switchboard/` directory and `.github/copilot-instructions.md` describe the Switchboard MCP protocol used for routing messages between agents across IDE instances. Inboxes and outboxes live in `.switchboard/inbox/<agent>/` and `.switchboard/outbox/<agent>/`. Workflows (handoff, improve-plan, challenge, accuracy) are defined in `.agent/workflows/`.
